Problem

AI 時代の構造的問題

Step 1

AI がコードを速く書く

設計書を書く前に実装が進む

Step 2

設計書が腐る

コードと設計書が乖離し始める

Step 3

次の AI が推測で動く

設計を読まずコードから推測し、さらに腐る

これは正のフィードバックループ。放置すると加速する。

Core Concept

整合性駆動開発

「設計書とコードが常に一致していること」を不変条件として維持する開発手法

  • docs 正本（設計書を先に書く）は手段であって目的ではない
  • 目的は整合性の維持。整合性が保たれている限り、AI は設計書を読んで安全に実装できる
  • 整合性が崩れた瞬間、AI は推測で動き始め、コードは設計から離れていく

spec-runner はこの不変条件を機械的に強制するハーネスである

Enforcement

不変条件の強制機構

maps_to 設計書とコードの対応を宣言する。パス推測に頼らない
scan.js + hooks maps_to の実在を Edit/Write のたびに検証。存在しないパスがあれば exit 1
@design-reviewer 責務・入出力・テスト観点の意味的整合を検証するエージェント
design-change フェーズゲートでコードを先に変えることを構造的に防ぐ
depends_on 変更の波及範囲を設計書の依存関係として宣言する
Declaration

frontmatter が整合性の宣言

---
spec_runner:
  node_id: detail.src.agents.reconciliation.agent
  kind: detailed_design
  depends_on:
    - overview.system_context
    - overview.use_case_list
  maps_to:
    - src/agents/reconciliation/agent.py
    - tests/agents/reconciliation/test_agent.py
---
maps_to

設計書とコードをつなぐ唯一の正本。scan.js がこの宣言を検証し続ける

depends_on

この設計書が何に依存するか。design-change が影響範囲を辿るときに使う

Key Insight

なぜプロジェクト専用スキルか

整合性はアーキテクチャに根ざしている。

DDD プロジェクト
  • 設計単位: 集約・値オブジェクト
  • 波及経路: ドメイン → UC → DB
  • テスト粒度: ドメインロジック単位
CRUD プロジェクト
  • 設計単位: UC・サービス層
  • 波及経路: UC → DB
  • テスト粒度: ユースケース単位

汎用スキルはアーキテクチャを知らないため、AI は「正しいフロー」を提供できず整合性の維持が形骸化する。
固定の skill pack を配るのではなく、決めたアーキテクチャから専用スキルを作る。

Design History — Era 1

シェルスクリプト強制型

やろうとしたこと

「AI も人間も絶対にスキップできない」フェーズゲートを外側から強制する

spec-runner.sh（1079行）
├── state.json でフェーズ管理
├── pre-tool-use.sh でコード出力をブロック
├── CI phase-gate-check.yml（216行）
├── grade-history.json（A/B/C/LOOP1）
└── UC ブランチ強制管理

何が無理だったか

  • 1079行のスクリプトはルール追加のたびに複雑化
  • 「絶対にスキップできない」を追求するほど穴が増えた
  • ブランチ強制が邪魔になるケースが多発
  • グレードシステムは特定チームにしか合わなかった

外側からの強制は機能しない

Design History — Era 2

Work Card 中心型

やろうとしたこと

シェルスクリプト強制を捨て、docs/work.md をシンプルな SSoT にする薄いオーケストレーターへ

  • grade-history・ブランチ強制・hooks を削除
  • work.md（目的・受入条件・タスク）
  • 再利用スキル uc-k1 / uc-k2

「薄くする」方向は正しかった

何が無理だったか

チェックリストは軽量すぎた。

  • 設計書との整合性を維持する仕組みがなかった
  • コードが設計から離れても検知できなかった
  • フェーズロックを個別修正で穴を埋め続けることになった

整合性を担保する手段がなかった

Design History — Era 3 (Current)

ハーネスエンジニアリング型

やったこと

CLI ランナーを完全に捨て、Claude Code にスキル・ルール・エージェントをインストールする道具に作り直す

削除: bin/spec-runner.js（CLI ランナー）
削除: docs/work.md
追加: skills / rules / agents テンプレート
追加: maps_to frontmatter + scan.js

何が変わったか

強制の方向が逆転した

外側からブロックするのではなく、設計書とコードの対応を宣言させ、その宣言を検証し続ける

Conclusion

思想の変化

Era 1

外側から強制する

「AI も人間もスキップできない仕組み」を押しつける

巨大化・硬直・穴だらけになる
Era 2

薄くして邪魔しない

チェックリストで軽量管理する

整合性を担保する手段がなくなる
Era 3 — 現在

宣言させ、検証し続ける

設計書とコードの対応を宣言させ、その宣言の実在を継続的に検証する

整合性駆動開発

強制から宣言へ。  外側からのブロックから内側からの不変条件へ。
