Core Concept

整合性駆動開発

「設計書とコードが常に一致していること」を不変条件として維持する開発手法

docs 正本（先に書く）は手段であって目的ではない。目的は整合性の維持。
整合性が崩れた瞬間、AI は推測で動き始め、コードは設計から離れていく。

Enforcement

不変条件の強制機構

maps_to設計書とコードの対応を宣言する。パス推測禁止
scan.js + hooksmaps_to の実在を Edit/Write のたびに検証。なければ exit 1
@design-reviewer責務・入出力・テスト観点の意味的整合をエージェントが検証
design-changeフェーズゲートでコードを先に変えることを構造的に防ぐ
depends_on変更の波及範囲を設計書の依存関係として宣言する
Declaration

frontmatter が整合性の宣言

---
spec_runner:
  node_id:
    detail.src.agents.reconciliation.agent
  kind: detailed_design
  depends_on:
    - overview.system_context
  maps_to:
    - src/agents/reconciliation/agent.py
    - tests/.../test_agent.py
---
maps_to が腐ると

scan.js が exit 1 を返す。設計書の腐敗をコミット前に検知。

design-change では

depends_on を辿り graph.json から波及範囲を自動列挙。

Key Insight

汎用スキルは
アーキテクチャを知らない

固定の skill pack を配るのではなく、人間と AI が決めたアーキテクチャから、そのプロジェクト専用の skill を作る。プロジェクト固有の構造を知ったスキルがあって初めて、整合性駆動開発は実務で機能する。

architecture.yaml → architecture-skill-development → プロジェクト専用 skills / rules

Design History

3つの失敗と1つの解

ERA 1 — 2026.03.12

シェルスクリプト強制

1079行のスクリプトで外側からブロック。「絶対スキップ不可」を目指した。

× 巨大化・穴だらけ
ERA 2 — 2026.03.20

Work Card 中心

work.md のチェックリストで薄く管理。強制を捨て軽量化を図った。

× 整合性担保なし
ERA 3 — 2026.04.12

宣言 + 継続検証

maps_to で宣言し scan.js で検証し続ける。CLI ランナーを完全廃棄。

✓ 整合性駆動開発
