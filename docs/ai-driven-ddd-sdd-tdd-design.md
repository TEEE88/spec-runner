# AI駆動 × 仕様駆動 × ドメイン駆動 × テスト駆動 一貫性設計

この文書は、`spec-runner` を使って「ドキュメントと実装が乖離しない開発フロー」を作るための設計方針を定義する。

## 1. ゴール

- 仕様（WHAT）と実装（HOW）の差分を、後追いではなく実行中に検出する
- AI の対話を活かしつつ、進捗・判断・未解決事項を構造化して残す
- ドメイン用語、受入条件、テスト、実装タスクを同じ流れでつなぐ

## 2. 非ゴール

- ブランチ運用の強制
- コミット粒度やコミット頻度の強制
- 単一ツールへの過度な依存（AIツールは将来差し替え可能にする）

## 3. 設計原則

1. 単一ソースの明確化  
   仕様・状態・成果物の「正本」を先に決める。
2. 生成より検証優先  
   AIは作るだけでなく、常に「仕様と一致しているか」を検証する。
3. ゲートは最小限  
   開発を止める条件は本当に必要なものだけにする。
4. 追跡可能性  
   要件 → 受入条件 → テスト → 実装タスクの対応を追えるようにする。

## 4. 正本（Single Source of Truth）

- フロー定義: `.spec-runner/steps/steps.json`
- 進行状態: `.spec-runner/phase-locks.json`
- 作業ラン記録: `docs/work.md`
- 仕様本文: `docs/01..06` と `docs/02_ユースケース仕様/**/UC-*.md`
- 実装検証: `tests/**` と `test_command.run`

## 5. コアユースケース（最優先）

### UC-1: 新規作業を開始する

- 前提: 新しい目的が入力される
- 操作: AIが `docs/work.md` を初期化し、最初の step を提案する
- 期待:
  - `docs/work.md` が作成される
  - 初回 step と理由が記録される

### UC-2: 仕様を対話で確定する

- 前提: 対象の UC 仕様が存在する
- 操作: AIが曖昧点を質問し、回答を仕様本文へ反映する
- 期待:
  - 未解決事項が減る
  - 更新した見出しと判断が `docs/work.md` に記録される

### UC-3: 仕様からテスト設計へ進む

- 前提: 仕様がレビュー可能な状態
- 操作: AIが受入条件をテストへ落とす
- 期待:
  - 仕様とテストの対応表が生成される
  - 「何を満たせば完了か」が明確になる

### UC-4: 実装後に乖離を検出する

- 前提: 実装変更がある
- 操作: AIが仕様・テスト・実装差分を照合する
- 期待:
  - 差分が「仕様不足 / 実装不足 / テスト不足」に分類される
  - 次アクションが提案される

## 6. 乖離を防ぐ仕組み

### 6.1 ラン記録（docs/work.md）

最低限、次を保持する。

- run メタ情報（開始時刻、目的、対象）
- step 履歴（返却 step、実施結果、失敗理由）
- 判断ログ（質問、回答、採用理由）
- マイルストーン（charter/uc/domain/arch/test/implement）
- コミット確認（提案のみ。実施は任意）

### 6.2 対応関係の可視化

仕様ごとに次の対応を記録する。

- Requirement ID -> Acceptance Criteria
- Acceptance Criteria -> Test Case
- Test Case -> Task / Code Path

### 6.3 差分チェック

1. 仕様変更があれば、影響テストを列挙  
2. テスト変更があれば、対応要件を逆引き  
3. 実装変更があれば、未更新仕様/未更新テストを検出

## 7. AI の役割分担

- Planner: 次の step と理由を提示
- Editor: ドキュメント/コードを更新
- Reviewer: 乖離チェックと不足の指摘
- Recorder: `docs/work.md` の更新

## 8. 導入順（段階導入）

Phase A（最小導入）
- `docs/work.md` 初期化
- step 履歴の記録

Phase B（一貫性導入）
- 要件-テスト-実装の対応表生成
- 乖離分類レポート

Phase C（運用最適化）
- 繰り返し作業の Skills 提案
- 技術スタック別のテンプレ分岐

## 9. 開いている論点（次に決める）

1. run の終了条件（実装完了時か、手動クローズか）
2. `docs/work.md` の最小必須セクション
3. レビュー時の表示形式（JSONのみ / JSON+Markdown）
4. 対応表をどのファイルに置くか（UC内 / runs配下）

---

この設計は「厳格な手順の固定」ではなく、「毎回の目的に合わせて同じ品質軸で進める」ための土台である。

## 10. シンプル運用ユースケース（docs/work.md 1ファイル）

この章は、上記方針を最小構成で回すための実運用ユースケースを定義する。  
正本は `docs/work.md` とし、次の4ブロックだけを必須にする。

- 目的
- 受入条件
- 実装タスク（チェックボックス）
- 検証結果

### UC-S1: 新規作業を開始する

- 前提:
  - 新しい作業目的がある
- 操作:
  - AI が `docs/work.md` を作成（既存がある場合は今回分に更新）
  - 「目的」と「受入条件」をユーザーと対話して確定
- 期待:
  - 何を達成するかが1画面で読める
  - 完了判定が曖昧でない

### UC-S2: 実装前に受入条件を固定する

- 前提:
  - 目的が記載済み
- 操作:
  - AI が受入条件を「テスト可能な文」で列挙
  - ユーザーが不足を追記・修正
- 期待:
  - 実装着手前に Done の定義が確定する
  - 仕様先行（SDD/TDD）を軽量に担保できる

### UC-S3: 作業中にタスク進捗を同期する

- 前提:
  - 実装タスクが `- [ ]` で列挙されている
- 操作:
  - AI が作業のたびに `- [x]` を更新
  - 新しく見つかった作業は追記
- 期待:
  - 実装実態とドキュメントが常に一致する
  - レビュー時に「何をやったか」が即座に分かる

### UC-S4: コミット前に未完了を検出する

- 前提:
  - コミット候補の変更がある
- 操作:
  - AI が `docs/work.md` の未完了チェック（`- [ ]`）を確認
  - 未完了があれば、コミット前に理由を明示して選択:
    - 完了させる
    - 今回は見送る（理由を `未解決` に記載）
- 期待:
  - 「タスク未完了のままコミット」を防げる
  - 例外運用でも意思決定が記録に残る

### UC-S5: 完了時に検証結果を固定する

- 前提:
  - 実装タスクが完了または見送り判断済み
- 操作:
  - AI が `検証結果` を埋める（テスト結果、手動確認、未解決）
- 期待:
  - 完了時点の品質証跡が残る
  - 後日、仕様と実装の乖離が起きても原因追跡が容易

### UC-S6: レビューで全体像を即時共有する

- 前提:
  - `docs/work.md` が最新
- 操作:
  - レビュアーは `docs/work.md` だけを読む
- 期待:
  - 目的、完了条件、作業内容、検証結果を短時間で把握できる
  - JSONや複数ファイルを辿らずに意思決定できる

## 11. 運用ルール（最小）

1. 実装開始前に「受入条件」を埋める
2. 作業中は「実装タスク」を更新する
3. コミット前に `- [ ]` が残っていないか確認する
4. 完了時に「検証結果」を必ず記入する

## 12. スキル化方針（最小）

スキル化は「繰り返し作業だけ」に限定する。  
最初から全部をスキル化しない。

### 方針

1. 2回以上同じ作業が発生したら候補にする
2. 候補は 1 スキル 1 目的（単機能）にする
3. スキルの入力と出力を固定し、自由記述を減らす
4. 失敗時のフォールバック手順を SKILL.md に必ず書く

### スキル化ユースケース

#### UC-K1: Work Card 初期化スキル

- 目的: `docs/work.md` の初期化
- 入力: 今回の目的（1文）
- 出力: 目的・受入条件・初期タスクを埋めた `docs/work.md`

#### UC-K2: コミット前チェック案内スキル

- 目的: `- [ ]` 未完了と検証漏れの検出
- 入力: なし（`docs/work.md` を読む）
- 出力: 未完了一覧と「コミットする/しない」確認文

#### UC-K3: 仕様-実装差分レビュー補助スキル

- 目的: 受入条件と実装タスクの不一致検出
- 入力: `docs/work.md` と差分ファイル一覧
- 出力: 不一致項目の一覧（仕様不足 / 実装不足 / テスト不足）

### 実行ルール

- スキル化は AI が提案し、ユーザーが採否を決める
- 採用後も `docs/work.md` を正本にする（スキルは補助）
- スキルが失敗した場合は通常手順（手動更新）へ戻す
