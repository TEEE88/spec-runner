# spec-runner 設計変遷履歴

整合性駆動開発という思想にたどり着くまでに何を試し、何が無理で捨てたかの記録。

---

## Era 1: シェルスクリプト強制型（2026-03-12 初回コミット）

### やろうとしたこと

「AI も人間も絶対にスキップできない」フェーズゲートを作る。

CLAUDE.md でルールを「知らせる」だけでは不十分という認識があった。Claude Code は知っていてもスキップできる。だからシェルレベルで実行を拒否する仕組みを作った。

```
spec-runner.sh（1079行）
  ├── state.json でフェーズ状態を管理
  ├── require / design-high / design-detail / test-design / implement ...
  ├── pre-tool-use.sh で implement フェーズ以外のコード出力をブロック
  ├── CI の phase-gate-check.yml（216行）でドキュメント鮮度を自動チェック
  └── grade-history.json で Grade A / B / C / LOOP1 を管理
```

UC ブランチも強制管理していた。UC ごとにブランチを切り、ブランチ名からフェーズを判定していた。slash commands は `sr-init`, `sr-design-high`, `sr-complete` など多数。

### 何が無理だったか

**スクリプトが巨大すぎて保守できなかった。**

1079行のシェルスクリプトは、ルールを追加するたびに複雑さが増した。「絶対にスキップできない」を追求するほど、穴が増えた。ブランチ強制は邪魔になるケースが多く、グレードシステムは特定チームの運用にしか合わなかった。「強制」を外側から押しつける設計の限界だった。

## Era 2: Work Card 中心型（2026-03-20）

### やろうとしたこと

シェルスクリプト強制を捨て、`docs/work.md` をシンプルな SSoT にする薄いオーケストレーターへ。

```
8d83447: refactor: simplify runner flow around work card
"Shift spec-runner to a thinner orchestrator centered on docs/work.md"
```

削除したもの:
- `grade-history.json`（グレード管理）
- `pre-commit / pre-push` hooks（ブランチ・フェーズ強制）
- UC ブランチ管理スクリプト群
- spec-runner-core.sh: 477行 → 279行

追加したもの:
- `docs/work.md`（目的・受入条件・タスクチェックリスト・検証結果）
- 再利用スキル `uc-k1-work-card-init`, `uc-k2-pre-commit-check`
- 設計ドキュメント `docs/ai-driven-ddd-sdd-tdd-design.md`

### 何が無理だったか

**`work.md` のチェックリストは軽量すぎた。**

設計書との整合性を維持する仕組みがなかった。何が設計書と対応しているかを宣言する方法がなく、コードが設計から離れていっても検知できなかった。UC discovery を先に飛ばしてしまう問題も残り、フェーズロック（`charter.completed` / `uc_discovery.completed`）を個別修正で埋め続けることになった。

「薄くする」方向は正しかったが、整合性の担保手段がなかった。

## Era 3: ハーネスエンジニアリング型（2026-04-12〜）

### やろうとしたこと

CLI ランナーとしての `spec-runner` を完全に捨て、Claude Code にスキル・ルール・エージェントをインストールする道具に作り直す。

```
0dcb4da: refactor: テンププレート構造刷新と命名規則の統一

削除:
  bin/spec-runner.js（CLIランナー 270行）
  docs/ai-driven-ddd-sdd-tdd-design.md
  docs/flow.md
  docs/work.md

追加:
  bin/spec-runner-installer.js（インストーラー）
  spec-runner/templates/.claude/skills/
  spec-runner/templates/.claude/rules/
  spec-runner/templates/.claude/agents/
  docs/ハーネスエンジニアリング構想.md（初登場）
```

整合性の担保手段として `maps_to` frontmatter と `scan.js` を採用した。設計書がコードのどのファイルに対応するかを宣言し、hooks で自動検証する。宣言が実在しなければ exit 1。

「強制」の方向が逆転した。外側からブロックするのではなく、設計書とコードの対応を**宣言させ、その宣言を検証し続ける**。

### 現在も続く改善（2026-04-14〜）

```
4ea98c2: feat: スキルにフロントエンド判定・architecture.yaml活用・UCカテゴリ分けを追加
```

- `has_frontend` — UC テンプレートをアーキテクチャに合わせて分岐
- `architecture.yaml` を設計の進行とともに更新し続ける契約として正式化
- UC のカテゴリ分けによる粒度の整理

## 捨てたものと残ったもの

| 概念 | 判断 | 理由 |
|------|------|------|
| シェルスクリプトによるフェーズ強制 | 捨てた | 巨大化・穴が増える。外側からの強制は機能しない |
| `grade-history.json` | 捨てた | 特定チームの運用にしか合わない |
| ブランチ強制 | 捨てた | 邪魔になる。コミットは人間と AI の合意で行う |
| `pre-tool-use.sh` によるコード出力ブロック | 捨てた | 強制より宣言と検証のほうが機能する |
| `docs/work.md` チェックリスト | 捨てた | 整合性の担保手段がなかった |
| CI `phase-gate-check.yml` | 捨てた | CI より hooks のほうがフィードバックが速い |
| フェーズロック（`charter.completed` 等） | 捨てた | 個別修正で穴を埋め続けることになった |
| `maps_to` frontmatter | 残した・強化した | 宣言ベースの整合性担保として機能している |
| `depends_on` frontmatter | 残した | 変更の波及範囲を宣言できる |
| `scan.js` + hooks | 残した | 宣言の実在を自動検証できる |
| フェーズゲート（ユーザー承認） | 形を変えて残した | スクリプト強制 → スキルのフェーズ管理へ |

---

## 思想の変化

**Era 1**: 「AI も人間もスキップできない仕組みを外側から押しつける」
→ 巨大化・硬直・穴だらけになる

**Era 2**: 「薄くして邪魔しない」
→ 整合性を担保する手段がなくなる

**Era 3**: 「整合性を宣言させ、その宣言を検証し続ける」
→ 現在の整合性駆動開発へ

強制から宣言へ。外側からのブロックから内側からの不変条件へ。
