速く作れる。でも保守できない。

AI 駆動開発の課題

7つの課題

① ドキュメントがなく保守不能修正のたびにコードを全部読む
⑤ AI に毎回前提を説明する毎セッション「このシステムは…」を繰り返す
② アーキテクチャが崩壊する共通化なし・ハードコード多発・責務が適当
⑥ コマンドを覚えて自分で打つ既存ツールは手動コマンド選択が必要
③ 設計判断が記録されないなぜこの設計にしたかが誰にもわからない
⑦ テストを後から書いても無意味既存挙動の確認にしかならない
④ 影響範囲がわからない修正後どこかでエラーが起きていることが多い
github.com/TEEE88/spec-runner
spec-runner とは

AI が設計を飛ばすことを技術的に防ぐシステム

スキル 12種 開発フローを定義。docs/ の有無で自動起動。プロジェクト専用スキルも自動生成
サブエージェント 3種 @test-runner（TDD 強制）/ @design-reviewer（設計⇔実装検証）/ @analyze-impact（影響算出）
ルール 2種 coding.md（命名・型・テスト規約）/ design-docs.md（設計書フォーマット・フェーズ管理）
scan.js + hooks Edit/Write のたびに maps_to の整合性を自動検証。ズレたら exit 1

Claude Code と GitHub Copilot の両方に対応。npx spec-runner 1コマンドで .claude/ と .github/instructions/ に同じルール・スキルを展開

解決 1 — 整合性の自動検証

設計書とコードを常に一対に保つ

maps_to 設計書 → 実装ファイルの対応を宣言
scan.js + hooks Edit/Write → 実在チェック → なければ exit 1
@design-reviewer 責務・入出力の意味的整合をエージェントが確認

設計書を変えずにコードを変えた瞬間、
自動で検知される

---
spec_runner:
  node_id: detail.usecase.注文確定
  kind: detailed_design
  depends_on:
    - overview.use_case_list
    - detail.domain.注文
  maps_to:
    - src/order/confirm.py    ← 宣言
    - tests/order/test_confirm.py
---

# Edit/Write のたびに自動実行
✓ src/order/confirm.py ... exists
✓ tests/order/test_confirm.py ... exists
# 存在しなければ exit 1 でブロック

解決 2 — 影響範囲の自動算出

「このUCを変えると何が壊れるか」を即答する

// .spec-runner/scan/graph.json（自動生成）
{
  "nodes": {
    "detail.usecase.注文確定": {
      "depends_on": ["overview.use_case_list"],
      "maps_to": ["src/order/confirm.py"]
    }
  },
  "reverse_index": {
    "overview.use_case_list": [
      "detail.usecase.注文確定",  ← 依存元
      "detail.usecase.注文一覧"
    ]
  },
  "missing_maps_to": []  ← ズレを記録
}
depends_on 宣言 上流の設計書への依存をfrontmatter に書く
scan.js Edit/Write のたびに graph.json を自動更新
@analyze-impact 変更 node_id → 直接・間接影響を reverse_index で即列挙
design-change 影響ドキュメントを確定してから次フェーズへゲート

解決 3 — プロジェクト専用スキルの自動生成

汎用スキルは土台。

  - usecase
  - infrastructure

解決 4 — TDD の「RED を目撃させる」

テストが失敗することを見ないと先に進めない

① RED
失敗するテストを書く
@test-runner 必須
失敗を目撃しなければ
次のフェーズへ進めない
② GREEN
通る最小コードを書く
@test-runner 必須
既存テストも
全部通ることを確認
③ REFACTOR
GREEN 後のみ
重複削除・命名改善
GREEN 後のみ
機能を変えない
リファクタのみ

最初から通るテスト = 既存挙動の確認にしかならない
→ @test-runner が判定し、テストを書き直させる

@test-runner テスト実行 + 失敗原因の分析まで自律実行
@code-reviewer GREEN 後に命名・型・規約を自動チェック
TDD スキル 実装コードを書く前のテスト作成を強制
