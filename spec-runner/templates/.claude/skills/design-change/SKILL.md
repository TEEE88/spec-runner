# design-change

フェーズは順番通りに進める。ユーザーの承認なしに先へ進めない。

設計記録テンプレートは `templates/02_概要設計/90_ADR/{mmdd}-{日本語タイトル}.md` を使用する。

## 前提: architecture.yaml の読み込み

開始前に `.spec-runner/architecture/architecture.yaml` を読み、以下を確認する:

| キー | 用途 |
|------|------|
| `scope` | `backend` / `frontend` / `all` — フロントエンドの設計書を変更対象に含めるか判断 |
| `folder_structure` | 影響ドキュメントのパス確認に使用 |

`scope` を確認したら、**このスキルファイルから scope 条件に合わないセクション（「scope: XX の場合はスキップ」と注釈されたセクション）を削除して最適化する**。

設計変更によって方針・構成に変化があった場合は、architecture.yaml も合わせて更新する。

## 全体フロー

```
Phase 1: 変更要求の整理と軽い影響調査
Phase 2: 設計記録作成（必要時）
Phase 3: 影響ドキュメントの確定
Phase 4: 概要設計の修正
Phase 5: ベストプラクティス調査（必要時）
Phase 6: 詳細設計の修正
Phase 7: TDD → 実装 → 検証
```

## Phase 1: 変更要求の整理と軽い影響調査

1. 変更の背景や目的について曖昧な点があれば、一問一答で深掘りする（各質問に推奨回答を添える。コードで確認できることは質問しない）
2. 以下をユーザーにヒアリングする
   - 変更の背景・課題
   - 変更の目的・期待する結果
   - 影響を受ける機能カテゴリ
   - バックエンド / フロントエンド / 両方のどちらに影響するか
   - 技術的・ビジネス的制約
3. 変更起点となる docs を特定する
4. `@analyze-impact` に委任して `depends_on` / `maps_to` を辿る影響ファイルを洗い出す
5. `scope: all` かつバックエンドの API（UC の入出力・エンドポイント）が変わる場合、フロントエンドの画面設計も影響対象に追加する（`docs/03_詳細設計/02_フロントエンド/01_画面/` の「API連携」セクションを確認）
6. 変更が影響するドメインと成果物を一覧化する
7. ユーザーに確認・承認を得る

## Phase 2: 設計記録作成（必要時）

1. アーキテクチャ、責務境界、保存方式、外部 IF などの意思決定が必要か判定する
2. 必要な場合は 3 案を提示する
3. 各案について概要・メリット・デメリット・適合性を示す
4. ユーザーが案を決定する
5. テンプレート `templates/02_概要設計/90_ADR/{mmdd}-{日本語タイトル}.md` をコピーして生成する
6. ファイル名は `mmdd-{日本語タイトル}.md`、配置先は対象フォルダ
7. ユーザーに確認・承認を得る

### 設計記録配置ルール

配置先・命名規則は `.github/instructions/design-docs.instructions.md` の ADR テーブルに従う。

## Phase 3: 影響ドキュメントの確定

1. `@analyze-impact` に委任して `depends_on` / `maps_to` を辿る影響ファイルを網羅的に洗い出す
2. ヘッダーの `depends_on` / `maps_to` を使って候補を絞る
3. 修正が必要なファイルをチェックリスト形式で提示する
4. `@review-design` エージェントに委任して、現時点の設計書⇔実装の乖離も合わせて確認する
5. ユーザーに確認・承認を得る

## Phase 4: 概要設計の修正

概要設計では「何をするか」に留める。比較案・採用理由・判断経緯は設計記録に置き、設計書本文へ持ち込まない。

### バックエンド（scope: frontend の場合はスキップ）

1. `docs/02_概要設計/01_システム全体設計/システム俯瞰図.md` / `システム構成図.md` を必要に応じて修正する
2. `docs/02_概要設計/02_バックエンド/ドメインモデル.md` / `業務ロジック概要.md` / `状態遷移図.md` を必要順に修正する

### フロントエンド（scope: backend の場合はスキップ）

1. `docs/02_概要設計/03_フロントエンド/画面一覧.md` / `画面遷移図.md` / `コンポーネント構成.md` を必要順に修正する

### インターフェース設計（scope: all かつ API 変更がある場合）

1. `docs/02_概要設計/04_インターフェース設計/API仕様.md` を修正する

修正完了ごとにチェックリストを更新する。全概要設計の修正完了後、ユーザーに確認・承認を得る。

## Phase 5: ベストプラクティス調査（必要時）

概要設計の修正が承認された後、詳細設計の修正に入る前に実施する。

1. 「詳細設計の修正に先立ち、ベストプラクティスを調査すべき技術・サービスはありますか？」とユーザーに確認する
2. 調査する技術・サービスの公式ドキュメント URL を `.spec-runner/references/resources.md` に追記する
3. WebSearch（resources.md の URL を起点に）でベストプラクティスを調査し、結果を保存する
4. 調査不要の場合はスキップして次へ進む

```
docs/04_調査資料/
  {カテゴリ名}/
    {トピック名}.md
```

ユーザーに確認・承認を得てから次へ進む。

## Phase 6: 詳細設計の修正

詳細設計本文にはコードを書かない。1 つの設計書には 1 つの主要責務だけを書き、`maps_to` でその責務を実装するコードとテストを対にして追跡する。

### バックエンド（scope: frontend の場合はスキップ）

ドメイン → UC → DB・外部サービス の順に修正する。

1. `docs/03_詳細設計/01_バックエンド/01_ドメイン/**` のドメイン設計を修正する
2. `docs/03_詳細設計/01_バックエンド/02_ユースケース/**` の UC 設計を修正する
3. DB・外部サービスの変更がある場合は `docs/03_詳細設計/01_バックエンド/03_DB・外部サービス/**` を修正する

### フロントエンド（scope: backend の場合はスキップ）

1. `docs/03_詳細設計/02_フロントエンド/01_画面/**` の画面設計を修正する
2. コンポーネントの変更がある場合は `docs/03_詳細設計/02_フロントエンド/02_コンポーネント/**` を修正する

ヘッダーの `depends_on` / `maps_to` を更新する。ユーザーに最終確認を得る。

## Phase 6.5: マイグレーション戦略（DB 変更がある場合のみ）

Phase 6 でスキーマ定義の変更があった場合に実施する。DB 変更がなければスキップして Phase 7 へ進む。

1. 既存の `docs/03_詳細設計/01_バックエンド/03_DB・外部サービス/DB/マイグレーション戦略.md` を確認し、今回の変更で影響する項目を洗い出す
2. 変更内容に応じて以下を更新する
   - Up / Down 方針の変更有無
   - ゼロダウンタイムの要否（既存データへの影響があるか）
   - ロールバック方針の変更有無
   - 既存データの変換が必要かどうか
3. マイグレーション戦略.md を更新する
4. ユーザーに確認・承認を得る

## Phase 7: TDD → 実装 → 検証

設計書修正が承認されたら `test-driven-development` スキルへ移行し、実装は `@implement-code` に委任する。
