# 実装

## 実行ルール

1. 変更要求から変更種別を判定する（振る舞い変更 / API 変更 / DB 変更 / 文言のみ）
2. 設計書の確認対象を決めて読む
   - 振る舞い変更・API 変更・DB 変更: `docs/02_概要設計/**` と `docs/03_詳細設計/**` を確認する
   - 文言のみ・内部リファクタ（外部仕様不変）: 関連する詳細設計だけ確認する
3. 設計と実装の差分がある場合の方針を決める
   - 設計どおりに実装する
   - 先に設計書を更新する（必要なら `review-design` / `analyze-impact` の手順を使って影響確認する）
4. `git diff --name-only` で変更対象を確認する
5. 変更領域に応じて規約を読む（共通ルール `.claude/rules/code-common.md` は両領域で必読）
   - バックエンド: `.claude/rules/code-backend.md` / `.claude/rules/test-backend.md`
   - フロントエンド: `.claude/rules/code-frontend.md` / `.claude/rules/test-frontend.md`
6. TDD 適用可否を判定する
   - 適用する: 新機能、振る舞い変更、バグ修正
   - 例外: 文言のみ変更、コメント整備、機械的リネームのみ
7. TDD を適用する場合は `test-driven-development` スキルに従って進める
8. 変更箇所に近い部分テストを先に実行する
9. 部分テストが成功したら全体テストを実行する
10. 失敗時は原因を特定して修正し、テストを再実行する

## テスト実行ルール

テスト実行コマンドは `.claude/rules/test-backend.md` / `.claude/rules/test-frontend.md` で定義されたコマンドを使用する。プロジェクトによって異なるため、実行前に必ず該当ルールファイルを確認する。

## 品質ルール

- `code-common.md` の規約（コメント・後方互換禁止・エラーハンドリング）を守る
- 仕様変更を伴う場合はテストを追加または更新する
- 想定外の失敗を握り潰さない
- 設計が正本の領域は実装だけ先行させない

## セルフチェック（報告前に必ず）

変更パスを対象に grep で残存リスクを確認し、見つかれば自分で直す。

- 後方互換キーワード: `後方互換|旧仕様|legacy|deprecated`
- 設計記録直接参照: `(ADR|adr|設計記録)\s*[0-9]`

部分テスト → 全体テストを実行し、`failed` が 0 であることを目で確認してから「成功」と報告する。

## バグ調査ルール（仮説より先に観測）

「再現するが原因が読めない」バグでは、コードを変更する前に事実を観測する。仮説で実装をいじって遠回りしない。

1. 再現条件を確定させる（曖昧なら `AskUserQuestion` で絞り込む）
2. 関係する箇所に観測ログ（呼び出し有無・ペイロード・応答 status / ヘッダ）を入れて 1 回再現させ、ログで読む
3. ログから事実を読み取り、想定値と外れた最初の地点を真因とする
4. 真因に対して最小限の修正を入れる（仮説段階の書き換え禁止）
5. 修正後にログを削除する（`git diff` で残存を確認）

外部ランタイム（フレームワーク・SDK）由来のバグは特に**まず観測**が最短経路。
