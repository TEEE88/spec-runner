# レビュー

コーディング規約チェックと設計書⇔実装の整合性チェックを行う。
呼び出し元から範囲指定があればそれに従う（コードのみ、設計書のみ、両方）。

## 1. コーディング規約チェック

### 実行ルール

1. `git diff --name-only` で変更ファイルを確認する（引数でファイルが指定された場合はそちらを使用）
2. `.claude/rules/code-common.md`（共通）と、変更対象に応じて `.claude/rules/code-backend.md` / `.claude/rules/code-frontend.md` を読み、チェック項目を把握する
3. 変更されたソースファイルを読み込む
4. 下記の優先順位で違反とリスクを確認する
5. 重要度順に報告する

### 優先順位

1. Security / Reliability
	- 認可漏れ、入力検証不足、秘密情報の露出、各種インジェクション、例外握り潰し、タイムアウト未設定
2. Performance
	- N+1、ループ内クエリ、不要な全件取得、過剰な同期 I/O
3. Regression / Testability
	- 既存機能への影響、境界条件、テスト不足
4. Dead code / Duplication / Backward-compat
	- 使われない引数・関数・分岐の残置
	- 共通化すべきローカル util の重複
	- 旧形式と新形式を両方サポートする `if-elif` 分岐
	- 設計記録 / 設計書名への直接参照コメント
5. Rule violations
	- コメント規約、命名、型、不要 import、`code-common.md` のエラーハンドリング規約

### 報告ルール

- 重大な問題だけを優先して書く
- 指摘ごとにファイル、行、問題、理由、推奨修正をセットで書く
- 問題がない項目を網羅的に列挙しない
- 指摘がない場合のみ「問題なし」を簡潔に返す

### コメント関連の扱い

`code-common.md` の「コメント」section を判定基準とする。短い目印コメントは違反扱いせず、違反として報告するのは: 逐語訳、冗長解説、誤った内容、設計記録への直接参照、セクション区切りコメント（`# === ===`）など。

## 2. 設計書⇔実装 整合性チェック

### 手順

1. `.claude/rules/design-docs.md` を読み、ヘッダー仕様（`node_id` / `kind` / `depends_on` / `maps_to`）と命名規則を把握する
2. `.claude/.spec-runner/scan/graph.json` を読み、`missing_maps_to` を取得する（`graph.json` は Edit / Write のたびに hooks で自動更新済み）
3. 監査対象の設計書ファイルを一覧取得する（メイン側から `docs/` のサブディレクトリ指定があればそれに従い、無ければ全件）
4. 各設計書のヘッダーを読み、`spec_runner.node_id` / `kind` / `depends_on` / `maps_to` を確認する
5. `docs/02_概要設計/**` の各ファイルを読む。`maps_to` から対応する `src/` ファイルを特定して読み、ユースケース・責務・外部 IF が実装に反映されているか比較する
6. `docs/03_詳細設計/**` の各ファイルを読む。`maps_to` に列挙された `src/` / `tests/` ファイルを実際に読み、責務・入出力・判断条件・テスト一覧を設計書と行レベルで比較する（`maps_to` が空または未設定の場合はヘッダーの問題として報告する）
7. 設計書本文のADR言及禁止ルールをチェックする
8. 乖離を報告する

### チェック項目

#### ヘッダー

- 参照原則: `maps_to` を唯一の src 対応とする。パス推定は行わない
- 必須項目: `node_id` / `kind` / `depends_on` / `maps_to` が存在するか
- 依存整合: `depends_on` の参照先が実在するか。依存の向きが明らかに逆転していないか
- 対応整合: `maps_to` の実ファイルが存在するか。変更対象が漏れていないか

#### 概要設計

- ユースケース整合: ユースケース一覧の各 UC が実装のエンドポイント・コマンド・画面遷移と一致しているか
- 全体俯瞰: システム俯瞰図の責務分割・連携フロー・外部 IF が実装構造と一致しているか
- ADR 反映: `docs/02_概要設計/90_ADR/**` の採用方針が概要設計・詳細設計に反映されているか

#### 設計書本文のADR言及禁止

`design-docs.md` のルールに従い、`docs/**/*.md`（ただしADR自身 `docs/02_概要設計/90_ADR/**` は除外）の本文に以下を含む記述があれば違反として報告する。設計書はADRから独立して読める内容にしなければならない。

- `ADR `〜` 参照` / `ADR-XXXX` への言及
- ADR ファイル名への参照
- `ADR `<コード>`` 形式の引用

検出方法: `grep -rn "ADR" docs/ --exclude-dir=90_ADR` を起点に、ADR番号 / ファイル名 / 「参照」セットで現れているケースだけを違反として報告する。`ADR` という単語が単独で出ているだけのケース（例: 「ADR を別途記録する」）は違反扱いしない。

報告形式:

```
### 設計書本文のADR言及（ルール違反）
- [ファイル]:[行番号]: [該当行の引用]
- 推奨対応: 採用後の仕様だけを設計書本文に書き、ADRへの参照は削除する
```

#### 詳細設計

- 責務整合: `docs/03_詳細設計/**` の責務記述が `maps_to` 先コードの責務と一致しているか
- 入出力整合: 関数シグネチャ、設定値、エラー処理、外部依存の扱いが設計どおりか
- テスト整合: `maps_to` に含まれる `tests/**` が設計上の主要シナリオをカバーしているか
- 不足 / 過剰: 設計書にあるがコードにない、またはコードにあるが設計書にない要素があるか

## 報告フォーマット

```
## レビュー結果

### コーディング規約チェック

#### 重大リスク（優先対応）
- [種別]
- [ファイルパス]:[行番号]
- [問題]
- [理由]
- [推奨修正]

#### 注意リスク（要確認）
- [種別]
- [ファイルパス]:[行番号]
- [問題]
- [理由]

#### 規約違反
- [ファイルパス]:[行番号]
- [規約]
- [内容]
- [修正案]

#### コーディング規約: 問題なし
- 重大な問題は見つからなかった

### 整合性チェック

#### ヘッダーの問題
- [ファイル]: [問題内容]

#### 一致（問題なし）
- [ファイルペアの一覧]

#### 乖離あり
##### [設計書ファイル] ⇔ [実装ファイル]
- 種別: 不足 / 過剰 / 変更
- 内容: [具体的な差分の説明]
- 推奨対応: 設計書を更新 / 実装を修正
```

ADRはコードと 1 対 1 の突合対象ではないが、配置先と反映有無は確認する。
