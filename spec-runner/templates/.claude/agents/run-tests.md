# テスト実行

## 実行ルール

1. `git diff --name-only` で変更ファイルを確認する
2. 変更ファイルに対応するテストファイルを特定する（`tests/` 配下の同構造）
3. 変更内容がバックエンドの場合は `.claude/rules/test-backend.md` を、フロントエンドの場合は `.claude/rules/test-frontend.md` を読む
4. まず変更箇所に近い部分テストを実行する
5. 部分テストが成功した場合、最後に全体テストを実行して回帰を確認する
6. 結果を分析する

## テスト実行コマンド（厳守）

テスト実行コマンドは `.claude/rules/test-backend.md` / `.claude/rules/test-frontend.md` で定義されたコマンドを使用する。プロジェクトによって異なるため、実行前に必ず該当ルールファイルを確認する。

ホスト環境のローカルツール（`pytest` / `npm test` / `npx playwright` など）を直接呼び出すコマンドが必要に見えても、プロジェクトのテスト実行方針（docker compose経由など）に従う。

## 優先順位

1. 変更箇所に直接対応するテストファイルがあれば最優先で実行する
2. 単体テストで十分なら単体テストを先に回す
3. 影響が複数層にまたがる場合だけ、結合テストや E2E を追加する
4. 最後に全体テストを実行して回帰を確認する
5. 対象テストが見つからない場合は、その不足自体を報告する

## 報告ルール

- 部分テストと全体テストを分けて報告する
- 失敗時は、どの段階で落ちたかを最初に書く
- 全体テスト未実施なら、その理由を明記する
- テスト不足がある場合は、失敗とは別に懸念として書く

## 報告フォーマット

```
## テスト結果

### 部分テスト
- 実行内容: [コマンドまたは対象ファイル]
- 結果: [成功 / 失敗 / 未実施]
- 補足: [必要なら要約]

### 全体テスト
- 実行内容: [コマンド]
- 結果: [成功 / 失敗 / 未実施]
- 補足: [必要なら要約]

### 失敗分析
- 失敗した段階: [部分テスト / 全体テスト]
- 失敗したテスト: [テスト名とファイルパス]
- エラー要約: [要約]
- 原因の推定: [推定]
- 修正案: [提案]

### テスト不足・懸念
- [対象テストがない / 回帰範囲が広い / 追加テストが必要 など]
```
