# コーディング規約（共通）

## コメント

人間レビューと 1 年後の追跡を前提に書く。

- 配置は基本コードの上。短い定数の補足のみ行末可
- ビジネス判断・制約・ワークアラウンドは必ず「なぜ」を書く
- 命名で分かることでも 1〜2 行の目印コメントは残してよい（自明コメントとして違反扱いしない）
- 逐語訳・超詳細解説・セクション区切り（`# === ===`）は禁止
- ファイル / モジュール先頭のコメントは不要（ディレクトリ構造で表現）
- TODO / FIXME マーカーは原則残さない（その場で対応するか削除する）。残す場合は何が必要かを 1 行で明記する
- 関数 docstring / JSDoc は公開 IF（UC、ドメインサービス、Repository、カスタムフック、ユーティリティ）のみ必須

## テスト記述

### 命名

- 設計書の `テスト一覧` の `T-XX` を関数名・ケース名に埋め込む
  - Python: `def test_T_01_refresh_false時は...():`（`-` は `_` に置換）
  - TypeScript: `test("T-01: refresh=false時は...", () => {})`
- 重複説明（コメント・docstring で ID/テスト名を書き直す）禁止
- 全テストケースで `# 準備 - ...` / `# 実行 - ...` / `# 検証 - ...` の構造化マーカーを必須とする。行数による免除はない（英語の Arrange/Act/Assert は禁止）
- このマーカーは規約由来であり、本ルールの「逐語訳・自明コメント」とは別物として扱う。レビューで違反指摘・削除提案の対象にしない

### ID 採番

- 形式: `T-XX`（2 桁ゼロ埋め）。100 件超のみ 3 桁
- スコープは設計書 1 ファイルごとに独立（別設計書と重複可）
- 設計書の `テスト一覧` の ID と関数/ケース名の ID を必ず一致させる
- `maps_to` で設計書 ↔ テストファイルを追跡

## 後方互換ハック禁止

使われないコードは完全削除。以下は禁止:

- 未使用の引数・変数を残す
- 削除した型・関数の再公開
- `# removed` / `# deprecated` / `# 旧実装` コメント付きで残置
- 後方互換用のラッパー / エイリアス
- 旧形式と新形式を `if-elif` で両方サポート

## エラーハンドリング

- 同じ抽出ロジック・型定義・変換ロジックを 2 箇所以上に書きそうになったら共通モジュールへ切り出す（ローカル util を量産しない）
- `ValueError` / `RuntimeError` のキャッチオールでステータスコードに変換しない。UC で意味のある例外にラップする
- 設計記録番号や設計書名への直接参照（`// ADR 0605 形式`、`# ADR-XXX 参照`）をコード・テスト・docs 本文に書かない
